# Global Knowledge-Node Schema v1

## Why This Layer Exists

The structural-parse pipeline now produces multiple valid thesis pointers to the
same underlying. Example:

- `MP` from `Rare Earth Magnet Sovereignty`
- `MP` from `Robotics Supply Chain`

Those should not remain as unrelated duplicate rows forever. They should resolve
onto one canonical knowledge node with multiple thesis pointers.

## Storage Model

v1 uses a hybrid structure:

- `thesis-local subgraphs`
  - preserved exactly as generated by each structural parse
  - keeps provenance and parse-specific context intact

- `global knowledge-node registry`
  - canonical nodes shared across parses
  - aggregates repeated entities like `MP`

- `bindings`
  - explicit links from parse-local entities onto canonical nodes

This keeps both:

- exact thesis provenance
- cross-parse memory and aggregation

## Canonical Node Schema

Each row in the registry is a canonical node.

### Core Fields

- `knowledge_node_id`
- `node_type`
- `canonical_key`
- `canonical_name`
- `aliases`
- `resolution_basis`

### Binding Fields

- `parse_entity_bindings`

Each binding stores:

- `parse_name`
- `structural_parse_path`
- `graduation_status`
- `graduation_score_0_to_100`
- `local_entity_id`
- `local_entity_type`
- `local_canonical_name`
- `local_attributes`

### Thesis Pointer Fields

- `thesis_pointers`

Each pointer stores:

- `pointer_id`
- `candidate_name`
- `market_theme`
- `thesis`
- `bottleneck_layer`
- `value_capture_layer`
- `promotion_status`
- `promotion_score_0_to_100`
- `company_role`
- `linked_process_layers`
- `linked_components`
- `linked_materials`
- `market_miss_alignment_score_0_to_100`
- `value_capture_alignment_score_0_to_100`
- `expression_readiness_score_0_to_100`
- `decomposition_confidence`
- `final_expression`
- `ranking_score`

### Aggregated View

- `linked_themes`
- `linked_process_layers`
- `linked_components`
- `linked_materials`
- `expression_views`
- `aggregate_view`

`aggregate_view` currently summarizes:

- `pointer_count`

- `themes`
- `expression_views`
- `max_ranking_score`
- `max_promotion_score_0_to_100`
- `top_pointer_id`

## Resolution Rules

v1 only resolves `Underlying` nodes.

### PublicCompany -> Underlying

Resolution key:

- `ticker:<SYMBOL>`

Rules:

- prefer ticker-like `canonical_name` when available
- uppercase symbol
- preserve parse-local attributes in bindings

Example:

- parse-local `PublicCompany MP`
- parse-local `PublicCompany MP`

Both resolve to:

- `knowledge_node_id = kn_underlying_mp`
- `canonical_key = ticker:MP`

### Future Node Types

The same pattern can later extend to:

- `Theme`
- `ProcessLayer`
- `Component`
- `MaterialInput`
- `Event`
- `Expression`

But v1 intentionally starts with `Underlying` because that is the highest-value
aggregation surface for picks.

## Parse-Local Mapping Model

The correct relationship is:

- parse-local entity stays local
- canonical node lives above the parse
- binding connects them

That means MiroFish keeps:

- thesis-specific graph meaning
- one-node memory for repeated entities

This avoids flattening away differences between:

- `MP` as a robotics-actuation exposure
- `MP` as a magnet-sovereignty exposure

while still recognizing that both point to the same listed company.

## Why This Matters

Without this layer, the system remains parse-centric.

With this layer, the system can start answering questions like:

- what are all the current theses pointing at `MP`?
- which process layers recur across those theses?
- does the system prefer stock or LEAPS across the full node?
- where are multiple promoted parses converging on the same underlying?

That is a much better substrate for real structural information arbitrage.
